# Baileys WhatsApp QR Setup

**If you already have wa-bridge on a VPS:** Set `WA_BRIDGE_URL` (e.g. `http://72.62.163.6:3001`) in `.env.local` and the CRM will use it for QR and status. You do **not** need to run the Baileys server below.

This guide is for running the **optional** Baileys server so users can connect their WhatsApp to the CRM by scanning a QR code (when not using wa-bridge).

## 1. Run the Baileys server

The CRM includes a standalone Baileys server in `scripts/baileys-server/`. It generates QR codes and manages per-user sessions (`user_<nocodb_user_id>_whatsapp_1`).

```bash
cd scripts/baileys-server
npm install
npm start
```

The server listens on **port 4050** by default. You can set `PORT=4051` to use another port.

Endpoints:

- **POST** `/connect?sessionId=user_<id>_whatsapp_1` — Start a connection. Returns `{ started: true }`.
- **GET** `/status?sessionId=user_<id>_whatsapp_1` — Returns `{ connected, phone?, qr? }`. Poll this after starting to get the QR (base64 data URL) or connected phone.
- **POST** `/api/logout-session` with `{ sessionId: "user_<id>_whatsapp_1" }` — Log out, destroy client, remove session.
- **POST** `/disconnect?sessionId=user_<id>_whatsapp_1` — Legacy disconnect endpoint.

Auth state is stored under `auth_user_<id>_whatsapp_1/` (or `BAILEYS_AUTH_DIR` when set) so reconnects do not require a new QR each time.

### Docker with persistent auth volume

```bash
cd scripts/baileys-server
docker compose up -d
```

The compose file mounts `baileys_auth_data:/data/auth` so sessions survive container restarts and VPS reboots.

## 2. Point the CRM at the Baileys server

In your Next.js `.env.local`:

```env
BAILEYS_API_URL=http://localhost:4050
```

If the Baileys server runs on another machine, use that URL (e.g. `http://192.168.1.10:4050`). The CRM will call this URL to start connections and poll for QR/status.

## 3. User flow in the CRM

1. User goes to **Dashboard → WhatsApp Connect**.
2. Clicks **Connect**.
3. A QR code appears (generated by Baileys).
4. User opens WhatsApp on their phone → **Linked Devices** → **Link a device** → scans the QR.
5. Once linked, the CRM shows the connected phone number. The same slot can reconnect later without a new QR (using saved auth).

## 4. Production notes

- Run the Baileys server on a stable host (same machine as the app or a dedicated server). Keep the `BAILEYS_AUTH_DIR` volume backed up so you don’t lose sessions.
- **Do not** use `useMultiFileAuthState` in high-throughput production; the script uses it for simplicity. For production, implement a custom auth state that writes to your database or cache (see [Baileys auth](https://baileys.wiki/docs/socket/connecting/#auth-state)).
- Expose the Baileys server only to your CRM backend (e.g. firewall, or run it locally and set `BAILEYS_API_URL=http://127.0.0.1:4050`).
